# RotarySlider Feature Studio & Visual Matrix Design

## Overview
This document outlines the architecture for the **RotarySlider Feature Studio**, a high-industrial, polyglot micro-frontend dashboard for monitoring and injecting intents into the recursive evolution matrix. It also defines the **Visual Matrix**, a closed-loop UI/UX generation pipeline that uses Art Gen (Stable Diffusion/DALL-E) to create assets and Vision Models (Claude 3.5 Sonnet) to visually gate and grade generated code.

## Section 1: The Polyglot Micro-Frontend Architecture
The Feature Studio is a decoupled web interface designed for extreme observability.
- **The Shell (Next.js + TypeScript)**: Acts as the primary orchestrator. Handles user intent injection, global state, and the overarching layout (sidebar, header).
- **The Telemetry Engine (SvelteKit)**: Embedded within the Shell. Dedicated exclusively to consuming a high-frequency WebSocket stream from the RotarySlider backend. Renders the infinite recursive epochs, hardware leasing states (CPU/VRAM), and tally matrices with zero-lag reactivity.
- **The Backend (FastAPI + MCP)**: A lightweight Python API wraps the existing `mcp_server.py` and `recursive_evolution.py` scripts, exposing REST endpoints for intent injection and WebSockets for telemetry.

## Section 2: The Visual Matrix (Art Gen & Visual V&V)
RotarySlider's Evolution Gate is expanding beyond mathematical/SLA checks to include visual perfection.
- **Asset Generation (Intent Phase)**: If a feature map includes a visual intent (e.g., "Generate a hero background image"), the swarm invokes an Art Gen API, downloads the asset, and commits it to the worktree.
- **Headless Rendering (Effect Phase)**: When the swarm writes frontend code (HTML/CSS/JS), the framework spawns a headless browser (Playwright) to render the component in isolation and capture a high-resolution screenshot.
- **Visual Gating (MaxVal Phase)**: The screenshot is piped to a Vision Model with strict UX heuristics (contrast ratios, alignment, padding, typographic hierarchy). The Vision Model returns a pass/fail grade. If it fails, the mutation is purged. If it passes, it merges.

## Section 3: Industrial Technicals & Boundaries
- **Strict Typing**: All boundaries between the Next.js shell, Svelte telemetry, and Python backend are governed by strict OpenAPI/JSON Schema contracts.
- **Hardware Piping**: The Visual Matrix (Playwright + Vision Models) is extremely resource-heavy. It will strictly abide by the `HardwarePipingManager`, leasing RAM and VRAM before spawning the headless browser or invoking local models.
- **Isolation**: The UI components generated by the swarm must be isolated in a "sandbox" directory to ensure the headless browser can render them without bringing down the main Feature Studio.
